## 1. Відповіді на контрольні питання

1. **Що таке REST API та які його основні принципи?**
REST API - це архітектурний стиль взаємодії між клієнтом і сервером через HTTP-протокол. Основні принципи: клієнт-серверна архітектура, відсутність збереження стану (Stateless), використання стандартних HTTP-методів, єдиний інтерфейс (Uniform Interface).
2. **Які HTTP-методи використовуються в REST API?**
GET (отримання даних), POST (створення ресурсу), PUT/PATCH (повне/часткове оновлення), DELETE (видалення).
3. **Що таке клієнт-серверна архітектура?**
Це поділ системи на дві незалежні частини: клієнт (відповідає за відображення та інтерфейс користувача) і сервер (відповідає за бізнес-логіку та роботу з базою даних).
4. **Що означає принцип Stateless у REST?**
Кожен запит від клієнта до сервера містить усю необхідну інформацію для його обробки. Сервер не зберігає інформацію про попередні запити клієнта (не зберігає його стан).
5. **Для чого використовується фреймворк Express у Node.js?**
Express - це мінімалістичний і гнучкий вебфреймворк, який значно спрощує маршрутизацію (routing), обробку HTTP-запитів і відповідей, а також інтеграцію проміжних обробників (middleware).
6. **Що таке авторизація та чим вона відрізняється від автентифікації?**
Автентифікація - це перевірка особи користувача (чи дійсно він той, за кого себе видає, наприклад, перевірка логіну та паролю). Авторизація - це перевірка прав доступу вже автентифікованого користувача до певних ресурсів.
7. **Для чого використовується JWT (JSON Web Token)?**
JWT використовується для безпечної передачі інформації між сторонами у вигляді JSON-об'єкта. Найчастіше застосовується для підтримки сесій (як токен доступу), оскільки він містить підпис і захищений від підробки.
8. **З яких частин складається JWT-токен?**
Він складається з трьох частин: Header (заголовок, містить тип токена та алгоритм шифрування), Payload (корисне навантаження, дані про користувача) та Signature (підпис для верифікації).
9. **Для чого потрібно хешування паролів і яка бібліотека використовується?**
Хешування необхідне для безпечного зберігання паролів у базі даних, щоб у разі витоку даних зловмисники не отримали паролі у відкритому вигляді. У роботі використовується бібліотека `bcryptjs`.
10. **Що таке middleware та яку роль він виконує в Express?**
Middleware (проміжне програмне забезпечення) - це функції, які мають доступ до об'єкта запиту (`req`), відповіді (`res`) і функції `next()`. Вони використовуються для перевірки токенів, логування, валідації тощо до того, як запит дійде до кінцевого контролера.

## 2. Структура вебзастосунку (Backend)

Файлова структура бекенд-сервера логічно розділена відповідно до патерну MVC (без View, оскільки це REST API).

**Організація файлової структури:**

- `server.js` - головний файл сервера, ініціалізація Express та підключення бази даних.
- `models/User.js` та `models/Task.js` - моделі Sequelize для бази даних.
- `models/index.js` - налаштування зв'язків між моделями (One-to-Many).
- `routes/authRoutes.js` та `routes/taskRoutes.js` - маршрутизатори для обробки кінцевих точок (endpoints).
- `controllers/authController.js` - бізнес-логіка реєстрації та авторизації.
- `middlewares/authMiddleware.js` - проміжний обробник для захисту маршрутів та перевірки JWT-токена.

## 3. Виконання завдань та програмний код

### 3.1. Реєстрація користувача, підтвердження пароля та хешування

У контролері `authController.js` реалізовано метод `register`. Дані валідуються (перевірка на порожні поля, довжину пароля, збіг пароля та його підтвердження). Пароль хешується за допомогою `bcryptjs` перед збереженням. Також реалізована перевірка на дублювання електронної пошти.

```javascript
// controllers/authController.js (фрагмент)
exports.register = async (req, res) => {
    try {
        const { name, email, password, confirmPassword } = req.body;

        if (!name || !email || !password || !confirmPassword) {
            return res.status(400).json({ message: "Всі поля обов'язкові" });
        }
        if (password.length < 6) {
            return res.status(400).json({ message: "Пароль мінімум 6 символів" });
        }
        if (password !== confirmPassword) { // Завдання: підтвердження пароля
            return res.status(400).json({ message: "Паролі не співпадають" });
        }

        const existingUser = await User.findOne({ where: { email } });
        if (existingUser) {
            return res.status(400).json({ message: "Користувач з таким email вже існує" });
        }

        const hashedPassword = await bcrypt.hash(password, 10);
        const newUser = await User.create({
            name,
            email,
            password: hashedPassword
        });

        res.status(201).json({ message: "Користувача успішно створено", userId: newUser.id });
    } catch (error) {
        res.status(500).json({ message: "Помилка сервера", error: error.message });
    }
};
```

### 3.2. Авторизація користувача (Login) та JWT

Метод `login` перевіряє правильність введених даних і, у разі успіху, генерує JWT-токен із терміном дії 1 година. У Payload токена також зашивається `id`, `email` та `role` (Завдання: роль користувача).

```javascript
// controllers/authController.js (фрагмент)
exports.login = async (req, res) => {
    try {
        const { email, password } = req.body;

        const user = await User.findOne({ where: { email } });
        if (!user) return res.status(400).json({ message: "Невірний email або пароль" });

        const isMatch = await bcrypt.compare(password, user.password);
        if (!isMatch) return res.status(400).json({ message: "Невірний email або пароль" });

        const token = jwt.sign(
            { id: user.id, email: user.email, role: user.role },
            process.env.JWT_SECRET,
            { expiresIn: "1h" }
        );

        res.json({ message: "Успішний вхід", token });
    } catch (error) {
        res.status(500).json({ message: "Помилка сервера", error: error.message });
    }
};
```

### 3.3. Захист маршрутів (Middleware)

Для захисту API розроблено проміжний обробник (middleware), який зчитує токен із заголовка `Authorization: Bearer <token>` і верифікує його. Цей middleware блокує неавторизовані запити (код 401).

```javascript
// middlewares/authMiddleware.js
const jwt = require('jsonwebtoken');

module.exports = (req, res, next) => {
    const authHeader = req.headers["authorization"];

    if (!authHeader || !authHeader.startsWith("Bearer ")) {
        return res.status(401).json({ message: "Немає токена, доступ заборонено" });
    }

    const token = authHeader.split(" ")[1];

    try {
        req.user = jwt.verify(token, process.env.JWT_SECRET);
        next();
    } catch (error) {
        res.status(401).json({ message: "Невірний або прострочений токен" });
    }
};
```

### 3.4. Моделі бази даних та зв'язки (Sequelize)

Користувачі та їхні завдання зберігаються у базі даних. Модель `User` за замовчуванням має роль `user`. Завдяки зв'язкам (`hasMany` / `belongsTo`), кожне завдання прив'язане до конкретного користувача через `userId`.

```javascript
// models/index.js
const User = require('./User');
const Task = require('./Task');

User.hasMany(Task, { foreignKey: 'userId', onDelete: 'CASCADE' });
Task.belongsTo(User, { foreignKey: 'userId' });

module.exports = { User, Task };
```

## 6. Висновки

У ході виконання лабораторної роботи було розроблено функціональний REST API на базі Node.js та Express із використанням ORM Sequelize.

Було реалізовано:

- Механізми реєстрації та авторизації за допомогою хешування паролів (`bcryptjs`).
- Валідацію вхідних даних (перевірка пароля та його підтвердження, перевірка унікальності email) та систему обробки помилок за допомогою HTTP-кодів стану.
- Надійний захист маршрутів застосунку (middleware) на базі JWT-токенів (`jsonwebtoken`).
- Збереження даних у реляційну базу даних із підтримкою ролей (`admin/user`) та зв'язком між таблицями (користувач -> завдання).

API успішно пройшло тестування через Postman, що підтверджує коректну обробку всіх передбачених сценаріїв: від реєстрації до керування ресурсами від імені авторизованого користувача. Відключення користувача (Logout) зазвичай реалізовується на стороні клієнта шляхом видалення збереженого токена доступу.
