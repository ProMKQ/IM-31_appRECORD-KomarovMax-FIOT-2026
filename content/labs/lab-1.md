## 1. Аналіз та моделювання системи

**Предметна галузь:** Система управління завданнями (Task Tracker). Інформаційна система призначена для організації робочого процесу, відстеження статусів виконання завдань та базового управління проєктами.

### Ключові складники опису системи

- **Актуальність теми:** У сучасному світі ефективне управління часом та завданнями є критично важливим. Розробка системи трекінгу завдань дозволяє оптимізувати робочі процеси та підвищити продуктивність.
- **Мета роботи:** Створення зручного та адаптивного вебзастосунку "Task tracker", який дозволяє користувачам переглядати, додавати та відстежувати статуси виконання завдань.
- **Об’єкт роботи:** Програмний продукт (вебзастосунок) для відстеження завдань, що розробляється.
- **Предмет роботи:** Властивості та характеристики процесів управління статусами завдань (pending, in progress, done) в інтерфейсі застосунку.

### Сценарій взаємодії (бізнес-логіка)

Бізнес-логіка описує сукупність правил, алгоритмів та обмежень, які визначають роботу системи.

1. Користувач відкриває застосунок та потрапляє на інформаційну панель (dashboard).
2. Система ініціалізує запит до сервера (`GET /tasks`) для отримання списку наявних завдань.
3. Користувач переглядає картки завдань, кожна з яких має відповідний статус виконання.
4. За потреби користувач може створити нове завдання, яке за замовчуванням отримує статус "pending".
5. У разі зміни етапу роботи, статус завдання оновлюється через API (`PUT /tasks/:id`).

### Функціональні вимоги

Функціональні вимоги описують, що саме повинна робити система.

- Система повинна відображати перелік завдань у вигляді інформаційних карток.
- Система повинна дозволяти отримувати (`GET`), створювати (`POST`), оновлювати (`PUT`) та видаляти (`DELETE`) завдання через API.
- Система повинна мати адаптивне навігаційне меню (бургер-меню для мобільних пристроїв).

### Нефункціональні вимоги

Нефункціональні вимоги описують якість роботи системи та її атрибути.

- Адаптивність: інтерфейс повинен коректно відображатися на десктопах, планшетах та смартфонах за допомогою медіа-запитів.
- Продуктивність: час завантаження та перебудови сітки контенту має бути мінімальним.
- Гнучкість: використання відносних одиниць вимірювання (`rem`, `%`) для забезпечення масштабованості.

## 2. Структура вебзастосунку

Файлова структура вебзастосунку логічно організована: HTML-документ, стилі та скрипти зберігаються окремо відповідно до їх призначення.

Організація файлової структури:

- `frontend/index.html` - головний структурний документ.
- `frontend/css/style.css` - таблиця стилів, що містить flexbox/grid сітки та медіа-запити.
- `frontend/js/app.js` - клієнтський скрипт для реалізації інтерактивності (бургер-меню).
- `backend/server.js` - Node.js/Express сервер для забезпечення бекенд-логіки.

### Структура сторінки `index.html`

Скелет сторінки складається із семантичних тегів: `<header>` з логотипом та навігацією, `<main>` із розташуванням карток завдань, та `<footer>` з довідковою інформацією.

```html
<!DOCTYPE html>
<html lang="uk">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Task tracker API</title>
    <link rel="stylesheet" href="css/style.css">
</head>
<body>
<header id="main-header">
    <div class="logo">TaskTracker</div>
    <nav id="nav-menu">
        <ul>
            <li><a href="#">Dashboard</a></li>
            <li><a href="#">My tasks</a></li>
            <li><a href="#">Settings</a></li>
        </ul>
    </nav>
    <div class="burger" id="burger-menu">
        <span></span><span></span><span></span>
    </div>
</header>

<main id="task-board">
    <section class="task-grid">
        <article class="task-card">
            <h3>Design database</h3>
            <p>Create ER diagrams for lab 1.</p>
            <span class="status pending">Pending</span>
        </article>
        </section>
</main>

<footer id="main-footer">
    <p>&copy; 2026 TaskTracker.</p>
    <p>Contact: support-tasktracker@gmail.com</p>
</footer>

<script src="js/app.js"></script>
</body>
</html>
```

## 3. Приклади реалізації елементів

У цьому розділі подано фрагменти коду та знімки з екрана, що демонструють реалізацію структурних компонентів інтерфейсу та функціонально-стилістичних рішень.

### 3.1. Адаптивна сітка контенту (CSS Grid)

Для карток завдань використано технологію CSS Grid з динамічною перебудовою залежно від ширини екрана.

```css
.task-grid {
    display: grid;
    grid-template-columns: repeat(auto-fit, minmax(250px, 1fr));
    gap: 1.5rem;
}

.task-card {
    background: white;
    padding: 1.5rem;
    border-radius: 8px;
    box-shadow: 0 4px 6px rgba(0,0,0,0.1);
    transition: transform 0.3s ease, box-shadow 0.3s ease; /* Анімація наведення */
}
```

### 3.2. Адаптивна навігація (Бургер-меню)

На пристроях із малою роздільною здатністю навігація трансформується у компактне "бургер-меню" за допомогою медіа-запитів.

```css
@media (max-width: 768px) {
    .burger { display: flex; }
    #nav-menu {
        display: none;
        position: absolute;
        top: 60px; right: 0;
        background-color: #2c3e50;
        width: 100%;
        text-align: center;
    }
    #nav-menu.active { display: block; }
}
```

```javascript
// app.js - Відкриття меню
const burger = document.getElementById('burger-menu');
const nav = document.getElementById('nav-menu');

burger.addEventListener('click', () => {
    nav.classList.toggle('active');
});
```
